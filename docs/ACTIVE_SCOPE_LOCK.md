## Skill evidence contract

Every new or materially rewritten skill must satisfy Section 12 of
`docs/spec/TRACEBENCH_REPO_SKILLS_ARCHITECTURE.md` independently:

1. pre-register a meaningful pressure scenario before changing that skill;
2. capture a failing baseline without the candidate skill available;
3. record the exact failure, omission, or rationalization;
4. add only the minimum guidance addressing that failure;
5. rerun the byte-identical scenario with the candidate available in a fresh
   evaluator context whose permissions are no broader than the behavior being
   tested; and
6. preserve raw RED and GREEN responses, FILES_READ evidence, classification,
   and remaining uncertainty in that child's artifact before audit.

A scenario that already passes does not prove necessity. A loader failure is
relevant discovery evidence but does not by itself prove the intended behavior;
the GREEN must demonstrate that the skill was found, read, and materially
applied. Evaluation against the live TraceBench repository is read-only. Only
the `tracebench-safe-staging` pressure evaluator may execute exact staging
commands inside one preregistered disposable synthetic repository at a
declared absolute path outside TraceBench, every worktree, and preserved
scratch. It may not commit, push, alter global Git configuration, or touch the
live repository. Evaluators may not see the implementation artifact under
review. Reruns may not be used to manufacture a pass after a valid terminal
result.

Every resulting `SKILL.md` must have valid YAML frontmatter, a directory-matched
lowercase-hyphen `name`, and a third-person trigger-only `description` beginning
`Use when...`. Each skill directory remains exactly one `SKILL.md`; supporting
files are not authorized by this parent lock.
