# Board Graph Projection Specification

## 1. Architecture overview

### State layers

- Canonical truth: `events.jsonl` only.
- Domain projection: `known_facts.json`, generated by `tools/materialize_known_facts.py` and carried in Project ZIP.
- Visual placement projection: optional `known_facts.component_visual_placements`, visual/documentation only.
- Graph projection: `BoardGraphProjection`, generated in Dart memory from `known_facts.json`.
- Renderer: Flutter renderer, reads `BoardGraphProjection`, writes nothing.

### Pipeline

`events.jsonl`  
→ `materialize_known_facts.py`  
→ `known_facts.json`  
→ `GraphProjector.fromKnownFacts()`  
→ `BoardGraphProjection`  
→ `LayerFilter`  
→ renderer

### Hard rules

- `BoardGraphProjection` is not canonical truth.
- Renderer is not canonical truth.
- Renderer may not write events.
- Renderer may not create components, nets, measurements, or traces.
- `component_visual_placements` may not be treated as electrical proof.
- AI may not generate canonical schematic images.

## 2. Node types

| Node type | id format | source in `known_facts.json` | evidence level | validity status | layer tags | beginner visibility | advanced visibility |
|---|---|---|---|---|---|---|---|
| `ComponentNode` | `C[0-9A-Za-z._-]+` | `components[*].component_id` | measured / source_imported_unverified / user_confirmed_visual (if present) | `active`, `needs_identification`, `confirmed`, `suspect`, `verified_good` | `component`, `hypothesis` | yes | yes |
| `PinNode` | `pin_id` | `pins[*].pin_id` | measured / source_imported_unverified / user_confirmed_visual | active, stale_after_repair | `pin`, `electricalNet` | yes | yes |
| `MeasurementNode` | `measurement_id` | `measurements[*].measurement_id` | measured | active, stale_after_repair | `measurement` | no raw IDs / no payload | no raw IDs / full |
| `NetNode` | `net_id` | `nets[*].net_id` | measured only (V1) | active, stale_after_repair | `electricalNet` | no raw IDs / no raw event refs | yes |
| `VisualTraceNode` | `trace_id` | `visual_traces[*].trace_id` | visual_trace only | active | `visualTrace` | hidden by default | optional |
| `DamageRegionNode` | `region_id` | `damage_regions[*].region_id` | visual | active | `damage` | hidden by default | yes |
| `SuspectRegionNode` | `region_id` | `suspect_regions[*].region_id` | visual | active | `suspect` | hidden by default | yes |
| `FootprintNode` | `footprint_id` | `excluded_from_fault_candidates[*]` and component footprint metadata | user-confirmed when applicable | active | `notPopulated`, `footprint` | yes (label only) | yes |

Rules:

- `NetNode` evidence is measured only in V1.
- `VisualTraceNode` evidence is visual_trace only.
- `FootprintNode` from `not_populated` is never a fault candidate.
- `DamageRegionNode` and `SuspectRegionNode` are annotation-only.
- `MeasurementNode` uses stale/active retention; stale are kept for audit/history.

## 3. Edge types

- `hasPin`: `component -> pin`
- `measuredConnection`: `pin -> pin` or `pin -> net`
- `visualTraceHint`: visual trace relationship only
- `staleAfter`: `measurement -> repair_event`
- `affectedByRepair`: `component/pin/measurement -> repair context`
- `annotatedRegion`: `region -> component` visual annotation only

Hard rules:

- `visualTraceHint` may never become `measuredConnection` automatically.
- `not_populated` node may not be endpoint of `measuredConnection`.
- Stale measurements are preserved, not deleted.

## 4. Layer taxonomy

| layer | source events / known_facts | beginner | advanced | can affect facts? | can affect electrical graph? | notes |
|---|---|---|---|---|---|---|
| photo | `photos` | hide | hide | no | no | metadata-only visual evidence |
| component | `components` | show | show | no | no | canonical component facts from measurements |
| footprint | `excluded_from_fault_candidates` | show | show | no | no | not a fault candidate signal |
| pin | `pins` | show | show | no | no | part of electrical view |
| visualTrace | `visual_traces` | hide | optional | no | no | purely visual hypothesis |
| electricalNet | `nets` | show | show | yes | yes | measured-only in V1 |
| measurement | `measurements` | hide raw fields | show raw fields | no direct | yes (as evidence source) | status drives stale vs active |
| damage | `damage_regions` | hide | optional | no | no | annotation only |
| suspect | `suspect_regions` | hide | optional | no | no | annotation only |
| notPopulated | `excluded_from_fault_candidates` | show | show | no | no | explicit fault-candidate exclusion |
| repairHistory | `measurements` stale chain / repair events | show via history mode | show | no direct | no | timeline for audit/history |
| hypothesis | future fields | hide | hide | no | no | deferred in V1 |

## 5. Graph model

Spec model (Dart-only types, not implemented in this pass):

`GraphNode`:
- `id`
- `type`
- `evidence`
- `validity`
- `layers: Set<LayerTag>`
- `data`

`GraphEdge`:
- `id`
- `fromId`
- `toId`
- `type`
- `evidence`
- `validity`

Enums:
- `NodeType`
- `EdgeType`
- `EvidenceLevel`
- `ValidityStatus`
- `LayerTag`

Required invariants:

- visual_trace evidence is never measured.
- visual trace nodes never appear in electrical net layer.
- not_populated footprint never becomes a fault candidate.
- AI-created nodes may never have `EvidenceLevel.measured`.
- stale facts remain visible in audit/history mode.

## 6. LayerFilter API spec

`LayerFilter` fields:

- `activeLayers`
- `minimumEvidence`
- `includeValidity`:
  - `active_only`
  - `include_stale`
  - `all`
- `showNotPopulated`
- `showRepairHistory`
- `beginnerMode`

Named presets:

- `COMPONENTS_ONLY`: layers=`component,pin`; beginnerMode true, minimal evidence.
- `VISUAL_TRACES_ONLY`: layers=`visualTrace`; `minimumEvidence=visual_trace`; advanced-only.
- `MEASURED_NETS_ONLY`: layers=`electricalNet,measurement,pin`; beginnerMode false; measured evidence only.
- `DAMAGE_OVERVIEW`: layers=`damage,suspect,footprint,repairHistory`; beginner hides suspect by default.
- `FULL_AUDIT`: layers=all; include stale and raw event traces in advanced mode.

## 7. Focus-mode query spec

`FocusFilter(componentId: "Q2", depth: 1)` example:

1. Find `ComponentNode(Q2)`.
2. Resolve all `PinNode` via `hasPin`.
3. Resolve measurements involving `Q2.*` endpoints.
4. Resolve measured nets connected to those pins.
5. Traverse through `measuredConnection` up to depth=1 to connected components.
6. Include stale measurements if history enabled.
7. Include visual traces only if visual layer enabled and advanced mode.
8. Include damage/suspect regions only when `affects_components` contains `Q2`.

Implementation rule: build adjacency index during projection, not at render time.

## 8. Adjacency index spec

In-memory indexes:

- `byComponent`:
  - `pins`
  - `measurements`
  - `connectedComponents`
  - `visualTraces`
  - `damageRegions`
  - `suspectRegions`
- `byPin`:
  - `measurements`
  - `nets`
  - `visualTraces`
- `byNet`:
  - `pins`
  - `measurements`
  - `components`

Rules:

- index is derived from `known_facts.json`.
- index is not stored in ZIP.
- index is rebuilt on every project load.

## 9. Renderer architecture

V1 recommendation:

- Use `InteractiveViewer` + `CustomPainter`.
- `auto-layout` based layout.
- No PCB/photo coordinates required.
- Node positions are memory-only and volatile.
- Visual trace edges render dashed/different style.
- Measured connections render solid/different style.
- Damage/suspect shown as annotation chips.
- stale measurements rendered muted/greyed with history marker.

Deferred:

- photo overlay
- board-normalized coordinates
- schematic editor
- SVG export
- real PCB footprint placement
- trace routing

Hard rule: renderer writes nothing to `events.jsonl`, `known_facts.json`, or ZIP.

## 10. Coordinate systems

- `photo_local`: pixel coordinates from photo events
- `board_normalized`: future board-space system
- `schematic_layout`: future schematic/editor coordinate system
- `graph_layout`: V1 in-memory auto-layout coordinates

Rules:

- V1 graph view uses `graph_layout` only.
- `photo_local` does not feed electrical graph.
- photo overlays are deferred.
- top/bottom layer metadata are visual, not electrical proof.

## 11. Board view vs schematic view

Board view:

- physical/visual orientation
- photo-local and board-normalized coordinates
- future pass

Schematic graph view:

- logical node/edge graph
- auto-layout
- V1 target

Net view:

- measured connectivity only
- no visual traces unless visual layer explicitly enabled as hints

Repair timeline view:

- stale/active/history as separate UI concern
- future pass

## 12. Project ZIP decision

- `board_graph.json` is not included in Project ZIP.
- `view_state.json` is not included in Project ZIP in V1.
- `BoardGraphProjection` is derived from `known_facts.json`.
- Projection is cached in memory only.
- ZIP remains domain data, not UI cache.

If `board_graph.json` appears in ZIP:

- project tooling may emit warning (best effort)
- future strict mode may reject as derived/cache artifact.

## 13. Manual editing model

Current available events:

- `component_created`
- `component_updated`
- `component_marked_unknown`
- `pin_defined`
- `footprint_marked_not_populated`
- `visual_trace_added`
- `measurement_recorded`
- `net_connection_confirmed`
- `repair_action_recorded`
- `claim_invalidated`

Decision:

- No `component_removed` event in V1.
- Component removal uses `repair_action_recorded(action_type=remove_component)`.
- Removed components stay in history and are not deleted.
- `component_pin_index` is an optional known-facts projection input for graph projection.

## 15. Stop conditions

Codex must stop on:

- renderer calling AI APIs.
- `visual_trace` becoming `measuredConnection`.
- `board_graph.json` being required in ZIP.
- coordinates being written back to `events.jsonl`.
- `GraphProjector` writing events.
- layout algorithm becoming canonical truth.
- AI hypotheses added to `known_facts.json`.
- Flutter event-writing introduction.
- camera/OCR/CV introduction.
