# FLUTTER_NEW_PROJECT_WIZARD_SPEC.md

## Purpose

- Define the V1 scope for creating a new local TraceBench project from Flutter UI.
- Keep implementation constrained to project bootstrap files and no new evidence facts.
- New project wizard follows the accepted empty-project tooling support from
  `TOOLS_EMPTY_PROJECT_SUPPORT_PASS`.

## V1 scope

- Allow user creation of a blank local project folder.
- Wizard creates only project-level files/folders and known required contract artifacts.
- No event writing, component/pin/measurement/net/photo creation, or repair action
  creation in V1.
- No inference, AI, OCR/CV, source-search, or hidden-layer fact promotion.

## Project-level files and manifest

### Required generated files/folders

- `manifest.json`
- `events.jsonl`
- `known_facts.json`
- `metadata/schema_versions.json`
- `exports/customer_report.md`
- `device_profiles/` (created/copy-resolved according to existing tooling path rules)

### Optional convenience folders

- `photos/` (may be created empty, optional)
- `notes/` (may be created empty, optional)

### `manifest.json` schema

- `project_id`: generated by Flutter in V1.
- `schema_version`: `"1.0"`
- `created_at`: current UTC ISO-8601 timestamp
- `device_type`: user input or `"unknown"`
- `model`: user input or `"unknown"`
- `symptom`: user input or `"not_provided"`

### Project ID

- Format: `prj_<8 lowercase hex chars>`
- Example: `prj_a3f2c891`
- Regex: `^prj_[a-f0-9]{8}$`
- User must not edit project_id in UI in V1.
- Project creation must fail if destination project folder already exists (collision
  prevention).

## Event/materialization behavior

- `events.jsonl` is created as an empty text file.
- No `project_created` event in V1.
- `known_facts.json` is never assembled directly in Dart.
- Wizard must invoke existing Python materializer path:
  - `tools/materialize_known_facts.py`
- Empty `events.jsonl` handling is provided by
  `TOOLS_EMPTY_PROJECT_SUPPORT_PASS` and must produce minimal valid `known_facts.json`.

## `metadata/schema_versions.json`

- Create at project creation time using existing default schema versions if the
  implementation can safely source them without duplicating tool-contract logic.
- If safe reuse is not possible in V1, route:
  `metadata/schema_versions.json` may be created at first import/export refresh path,
  documented as an accepted temporary implementation detail.

## `exports/customer_report.md` placeholder

- Create placeholder file on new project creation.
- Placeholder must remain evidence-neutral, no diagnosis/fault claims.
- Recommended text:

```md
# TraceBench Project Report

Project: <project_id>

No customer report evidence has been recorded yet.
```

## Folder load expectations

- New project should be valid for existing import/open paths that rely on local folder
  contents + manifests.
- Initial `ProjectState` (when loaded) should be:
  - `isProjectionStale == false`
  - empty events
  - empty known-facts collections
  - placeholder report loaded
  - local `projectDirectory` set to created folder if loader supports folder-backed
    projects.

## Project ZIP compatibility

- New project must be exportable by existing `tools/export_project_zip.py`.
- Exported ZIP should pass `tools/validate_project_zip.py`.
- No `board_graph.json` / `view_state.json` generation or inclusion.

## Mobile and platform behavior

- Desktop/dev path is primary.
- Mobile-specific storage/path expansion implementation is **not** included in this
  scope-lock.
- If picker is not scoped on mobile, wizard may be desktop-only in V1 or
  show a placeholder.

## Forbidden facts/events in wizard V1

- component_id / pin_id / net_id creation
- measurements
- photos or visual trace facts
- visual-trace to electrical net promotion
- repair actions
- `events.jsonl` entries beyond empty bootstrap file
- `project_created` event

## UI fields (V1)

- Required:
  - destination folder
- Optional:
  - `device_type`
  - `model`
  - `symptom`

### Required interactions

- create button
- cancel/back

### Success behavior

- either load created project directly (if safe within current loader contract), or
  return to overview with explicit user-visible success.

## Future scope decisions

- folder-based UX improvements
- richer metadata collection and validation
- mobile implementation
- project import smoke test automation
- desktop runner/platform hardening

## Required test matrix for `FLUTTER_NEW_PROJECT_WIZARD_PASS`

- project_id format matches regex
- unique folder/collision handling
- manifest required values + defaults
- created_at is UTC ISO-8601
- empty `events.jsonl` created
- materializer invoked for empty-project known facts generation
- `known_facts.json` collections/maps empty after bootstrap
- placeholder report text is neutral (no claims)
- `device_profiles/` presence
- new-project zip export succeeds
- zip validation succeeds
- imported new project loads
- `isProjectionStale` is false after creation
- invalid folder rejected
- no component/pin/measurement/net/repar action events are created
- optional UI fields can be blank and default correctly
- mobile/desktop-only routing behavior is documented and tested if implemented
